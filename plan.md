# 番茄钟 Chrome 插件 - 开发计划

## 核心理念

**解决"看到了但不想停"的问题**，而非做又一个番茄钟。所有设计围绕"让停下来变得容易，让拖延变得昂贵"。

---

## 一、技术栈选型

| 层次 | 选型 | 理由 |
|------|------|------|
| 扩展规范 | Manifest V3 | Chrome 强制要求，新项目无理由用 V2 |
| 后台 | Service Worker | MV3 架构要求，需处理计时器持久化 |
| 前端 | 原生 HTML/CSS/JS | 插件体积小、启动快、无需构建工具即可运行 |
| 存储 | `chrome.storage.local` | 状态、配额、统计数据 |
| 通知 | `chrome.notifications` + 覆盖层注入 | 双通道：系统通知 + 页面覆盖层 |
| 脚本注入 | `chrome.scripting.executeScript` | 向活动标签页注入休息锁定覆盖层 |
| 图标/字体 | SVG + 系统字体栈 | 无依赖、高清 |

## 三、状态机设计

计时器的核心是一个明确的状态机，所有 UI 和行为都由它驱动：

```
IDLE ──start──> FOCUSING ──time-up──> REFLECTING ──submit──> BREAKING ──time-up──> IDLE
                    │                      │                     │
                  reset                  reset              (不可关闭)
                    ↓                      ↓
                  IDLE                   IDLE
```

- **IDLE**：未开始
- **FOCUSING**：专注中（25 min，可配置）
- **REFLECTING**：时间到，等待用户输入"完成了什么"（强制元认知切换）
- **BREAKING**：休息锁定中（5 min，期间覆盖层不可关闭）
- **EXTENDED**（子状态）：用户点"再 5 分钟"后的延长专注（消耗配额）

## 四、开发里程碑

### M1｜基础骨架（1-2 天）
**目标**：能启动、能计时、能通知，完成最小可用原型。

- [ ] `manifest.json`：声明权限（`storage`、`notifications`、`alarms`、`scripting`、`activeTab`、`<all_urls>`）
- [ ] Service Worker 基础框架：用 `chrome.alarms` 而非 `setTimeout`（MV3 下 SW 会休眠）
- [ ] Popup 界面：开始 / 暂停 / 重置 / 跳过 四个按钮 + 倒计时显示
- [ ] 状态持久化：SW 重启后能从 `chrome.storage` 恢复计时
- [ ] 系统通知：时间到时弹窗

**验收**：能完整走完 25 分钟 → 通知 → 5 分钟休息 → 通知 的循环。

---

### M2｜强制休息锁定（2-3 天，核心价值）
**目标**：时间到后，所有标签页被休息覆盖层占据，无法正常工作。

- [ ] Content Script 注入机制：休息开始时，向所有可见标签页注入 `lockscreen.js`
- [ ] 覆盖层 UI：全屏半透明背景 + 居中倒计时 + 鼓励文案
- [ ] **关闭的三次确认**：要强行退出，必须在 10 秒内连续点击"确认退出"三次，且每次文案不同（"真的要跳过？" → "今天已经跳过 N 次" → "确定放弃这次休息"）
- [ ] 新开标签页的处理：监听 `chrome.tabs.onCreated`，新标签页也要立刻注入
- [ ] 黑名单/白名单：某些域名（如银行、视频会议）可配置不注入，避免误伤

**验收**：休息中无论切换多少标签页、开多少新窗口，都无法绕过覆盖层。

**风险点**：
- iframe、受限页面（`chrome://`、Chrome 商店）无法注入——在这些页面时需用全屏通知兜底
- 用户可能直接禁用插件——这不是技术问题，是设计哲学问题，不做技术对抗

---

### M3｜有限续杯（1 天）
**目标**：允许"再 5 分钟"，但有配额。

- [ ] Popup 显示剩余配额："今日剩余 2/3 次"
- [ ] 点"再 5 分钟"消耗一次配额，计时器从 FOCUSING 进入 EXTENDED 子状态
- [ ] 配额用尽后按钮置灰，文案改为"今日配额已用完"
- [ ] 每日 0 点重置配额（用 `chrome.alarms` 的周期任务）
- [ ] 设置页可配置每日上限（默认 3 次）和每次延长时长（默认 5 分钟）

**验收**：连续 4 次点延长，第 4 次按钮不可用。

---

### M4｜强制元认知输入（1 天）
**目标**：时间到时弹窗问"这一番茄你完成了什么？"，打字后才能进入休息。

- [ ] REFLECTING 状态 UI：独立的全屏页面（而非小弹窗，视觉上必须打断）
- [ ] 输入框至少 10 个字符才能提交（防止糊弄式"完成了"）
- [ ] 记录到 `chrome.storage.local`，形成当日完成事项列表
- [ ] 跳过选项：有，但要点"跳过并说明原因"，且记入统计

**验收**：不打字或打字太短，"开始休息"按钮保持禁用。

---

### M5｜反向激励系统（2 天）
**目标**：奖励"按时停下来"，不奖励"做了多少番茄"。

- [ ] 统计维度（核心是前三个）：
  - 连续按时休息天数（streak）
  - 当日按时停下比例（完成数 / 完成数 + 跳过数）
  - 续杯克制度（今日未用完的配额越多越好）
  - ~~番茄总数~~（刻意不展示，避免成为 KPI）
- [ ] 徽章系统：连续 3/7/14/30 天按时休息解锁
- [ ] 小图鉴：每个徽章对应一个像素风小图（SVG），Popup 底部展示
- [ ] 在 Popup 首屏用一行话呈现："你已连续 5 天按时停下来 🌱"

**验收**：连续 3 天完成休息后，徽章出现；跳过任意一天，streak 归零并显示"从头再来不丢人"。

---

### M6｜设置页与打磨（1-2 天）
- [ ] 专注时长 / 休息时长 / 每日续杯次数 / 每次续杯时长 可配置
- [ ] 黑白名单管理 UI
- [ ] 导出数据（JSON）功能
- [ ] 快捷键支持（`chrome.commands`）：开始 / 暂停 / 跳过
- [ ] 图标多态：IDLE / FOCUSING（带进度角标）/ BREAKING 三种状态的图标

## 五、关键技术决策

1. **计时源用 `chrome.alarms` + 时间戳**，不用 `setInterval`。Service Worker 会被 Chrome 随时休眠，只有 alarms 和持久化的时间戳组合才能保证计时准确。
2. **状态只存一份，在 Service Worker 中**。Popup 和 Content Script 通过 `chrome.runtime.sendMessage` 查询和订阅，避免状态不一致。
3. **Content Script 的覆盖层用 Shadow DOM**，防止被页面 CSS 污染，也防止影响原页面样式。
4. **"三次确认退出"的实现**要避免用户脚本连点——相邻两次点击间隔必须 > 300ms，防止被脚本或快速连点绕过。

## 六、不做什么（范围界定）

- ❌ 跨设备同步（`chrome.storage.sync` 有 100KB 限制，后期再说）
- ❌ 账号系统 / 云端数据
- ❌ 任务管理（番茄钟只管时间，不管事项列表——那是另一个 App）
- ❌ 白噪音 / 音乐播放（容易喧宾夺主）
- ❌ 番茄总数排行榜（违背产品哲学）

---

## 七、开发顺序建议

按 M1 → M2 → M4 → M3 → M5 → M6 的顺序推进。

**理由**：M2（强制锁定）和 M4（元认知输入）是产品的灵魂，优先验证这两个机制是否真的有效——如果 M2 做完体验下来发现太狠反而被禁用，整个产品方向就需要调整。M3、M5 是锦上添花。
